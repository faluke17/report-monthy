1. MNF คืออะไร
----------------------------------------------------------------

MNF = Minimum Night Flow = ปริมาณน้ำไหลน้อยสุดในช่วงกลางคืน (ประมาณ 02:00–04:00)
เป็น indicator หลักของน้ำรั่วใต้ดิน เพราะช่วงกลางคืนการใช้น้ำของผู้ใช้เกือบเป็นศูนย์
ถ้า MNF สูง = น้ำไหลออกจากท่อโดยไม่มีคนใช้ = รั่ว

วัดผ่าน Logger ที่ติดตั้งตาม DMA (District Metered Area) แต่ละ node


----------------------------------------------------------------
2. Data Flow ทั้งหมด
----------------------------------------------------------------

[DMAMA API] → mnf_daily → EMA คำนวณ → mnf_ema_daily → mnf_ema_latest (view) → หน้าจอ

อธิบายแต่ละขั้น:

2.1 DMAMA API (https://dmama.pwa.co.th/api)
    - ระบบ NRW กลางของ กปภ.
    - Login ด้วย username/password รับ access_token
    - เรียก endpoint: /report/flow_rate_in_mnf
    - ส่ง parameter: branch_id, from, to, frequency=1440
    - Response: dataset ของแต่ละวัน แต่ละ logger
    - ข้อมูล: mnf_flow (m³/hr), min_pressure, mnf_at (เวลาที่วัด)

2.2 mnf_daily (ตาราง Supabase)
    - เก็บ raw data จาก DMAMA
    - Primary key: dmama_branch_id + logger_id + record_date
    - ใช้ upsert — รันซ้ำได้ไม่มีปัญหา
    - มี fetched_at บอกว่าดึงมาเมื่อไหร่

2.3 Cron Schedule (Vercel)
    - รอบ 1: 10:00 BKK (03:00 UTC) ทุกวัน
    - รอบ 2: 14:00 BKK (07:00 UTC) ทุกวัน — fallback ถ้ารอบแรก fail
    - mode=daily: ดึงข้อมูลเดือนปัจจุบัน (เพื่อให้ได้ข้อมูลเมื่อวาน)
    - retry: login 3 ครั้ง, แต่ละสาขา 3 ครั้ง

2.4 EMA คำนวณ (lib/utils/ema-calc.ts)
    - หลังดึงข้อมูลเสร็จจะ recompute EMA อัตโนมัติ
    - ดึง mnf_daily ย้อนหลัง 60 วัน (warm-up) + ช่วงที่ต้องการ
    - จัดกลุ่มตาม (dmama_branch_id, logger_id) แต่ละ node คำนวณแยก
    - filter node ที่ไม่น่าเชื่อถือออกก่อน (ดูข้อ 3)
    - upsert ผลลัพธ์ลง mnf_ema_daily ทีละ 500 rows

2.5 mnf_ema_latest (view)
    - View ที่ดึง row ล่าสุดของแต่ละ node จาก mnf_ema_daily
    - หน้าจอใช้ view นี้แสดงสถานะปัจจุบัน

2.6 หน้าจอ MNF Monitor (/mnf-monitor)
    - Region: grid 26 สาขา เรียงจาก worst ก่อน คลิกสาขาดูกราฟ
    - Branch: เห็นแค่สาขาตัวเอง ดูกราฟ 30 วันได้เลย
    - มุมขวาบน: sync status badge บอกว่าข้อมูลล่าสุดวันไหน

3. Node Classification — กรองก่อนคำนวณ
----------------------------------------------------------------

ทุก node ผ่านการกรอง 5 ชั้นก่อนจะคำนวณ EMA
ถ้า skip = node นั้นจะไม่ปรากฏในตารางและไม่มี alert

ชั้น 1: null เยอะเกิน (sparse)
    - ถ้า null > 60% ของข้อมูลทั้งหมด → skip
    - เหตุผล: logger เสียหรือออฟไลน์บ่อย ข้อมูลไม่น่าเชื่อถือ

ชั้น 2: ออฟไลน์อยู่ตอนนี้ (offline)
    - ถ้า 7 วันล่าสุดเป็น null ทั้งหมด → skip
    - เหตุผล: logger ปิดอยู่ ตีเป็นแดงไม่ได้

ชั้น 3: node เล็กเกิน (tiny)
    - ถ้า median < 1.0 m³/hr → skip
    - เหตุผล: node ขนาดจิ๋ว fluctuation ตามธรรมชาติก็เกิน threshold แล้ว

ชั้น 4: bimodal (สองหน้า)
    - ถ้า q1 < median × 10% → skip
    - เหตุผล: node มีช่วงยาวที่แทบไม่มีน้ำไหล (วาล์วปิด / ไม่ได้ใช้)
              EMA จะถูก distort ไม่น่าเชื่อถือ

ชั้น 5: normal
    - ผ่านทุกชั้น → คำนวณ EMA ปกติ

4. EMA Algorithm
----------------------------------------------------------------

สูตร:
    EMA_วันนี้ = MNF_วันนี้ × 0.1333 + EMA_เมื่อวาน × 0.8667

    (EMA-14: multiplier = 2 / (14+1) = 0.1333)

ความหมาย:
    EMA คือค่าเฉลี่ยถ่วงน้ำหนักที่ให้ความสำคัญวันใหม่มากกว่าวันเก่า
    แต่ยังจำ "pattern ปกติ" ของ node นั้นอยู่
    ใช้เป็น baseline เปรียบเทียบ — "ปกติ node นี้ไหลเท่าไหร่"

กฎการคำนวณ:
    - วันแรก: EMA = MNF วันนั้น, diff=0, status=green
    - null หลัง init: EMA คงเดิม, diff=0, status=green
    - วันปกติ: คำนวณตามสูตร

diff_percent:
    diff% = (MNF_วันนี้ - EMA_เมื่อวาน) / EMA_เมื่อวาน × 100
    บวก = MNF สูงกว่าปกติ (อาจรั่ว)
    ลบ = MNF ต่ำกว่าปกติ (ปกติ หรือวาล์วปิด)


----------------------------------------------------------------
5. Alert Logic
----------------------------------------------------------------

4 ระดับ:

green          diff% < 50%                        ปกติ
yellow         diff% ≥ 50%                        เริ่มสูง เฝ้าดู
red_spike      diff% ≥ 200% (ทันทีวันเดียว)       ท่อแตกฉุกเฉิน
red_accumulated consecutive_count ≥ 3 วัน          รั่วสะสมเรื้อรัง

consecutive_count:
    - เพิ่ม 1 ทุกวันที่ diff% ≥ 50%
    - reset เป็น 0 ถ้าวันไหนต่ำกว่า 50%


----------------------------------------------------------------
6. จุดอ่อนที่รู้อยู่ (ยังไม่ได้แก้)
----------------------------------------------------------------

6.1 รั่วช้าๆ ตรวจไม่เจอ (ปัญหาหลัก)
    EMA-14 ปรับตัว 13% ต่อวัน
    ถ้า MNF ค่อยๆ ขึ้นทีละนิด EMA ไล่ตาม diff% ไม่เคยถึง 50%

    ตัวอย่าง:
    วันที่ 1:  MNF=5.5, EMA=5.1, diff=+8%   → เขียว
    วันที่ 5:  MNF=7.5, EMA=5.9, diff=+27%  → เขียว
    วันที่ 10: MNF=10,  EMA=7.2, diff=+39%  → ยังเขียว!
    รั่วมา 10 วัน ไม่มี alert

6.2 Boiling Frog — EMA เรียนรู้ค่ารั่วเข้าไปใน baseline
    ถ้ารั่วมา 2 เดือนแล้ว EMA ได้ "ยอมรับ" ระดับนั้นว่าปกติ
    พอเกิดรั่วใหม่เพิ่ม ระบบเทียบกับ baseline ที่สูงแล้ว → alert ไม่ขึ้น
    เป็นปัญหาที่ทำให้ระบบมีปัญหาระยะยาว

6.3 consecutive reset ง่ายเกินไป
    แค่วันเดียวที่ diff < 50% นับใหม่จาก 0
    yellow 2 วัน / เขียว 1 วัน / yellow 2 วัน → ไม่เคยเป็นแดง
    ทั้งที่มีปัญหาชัดเจน

6.4 threshold เดียวกันทุก node
    50% ของ node เล็ก (1 m³/hr) = น้ำเพิ่ม 12 m³/วัน
    50% ของ node ใหญ่ (30 m³/hr) = น้ำเพิ่ม 360 m³/วัน
    ทั้งสองขึ้น yellow เหมือนกัน ความรุนแรงต่างกันมาก


----------------------------------------------------------------
7. สรุป ตรวจจับอะไรได้บ้าง
----------------------------------------------------------------

ได้:
    ✅ ท่อแตกฉับพลัน (MNF พุ่ง 3× ขึ้นไป)
    ✅ รั่วกะทันหันปานกลาง (MNF +80% ใน 1-2 วัน)

พลาด:
    ❌ รั่วค่อยๆ สะสม (ขึ้นวันละนิด)
    ❌ รั่วเรื้อรังที่มีมานาน (EMA ปรับตัวแล้ว)
    ⚠️  รั่วเป็นช่วงๆ สลับหาย (consecutive reset)

8. สิ่งที่แก้ไปแล้วในรอบนี้
----------------------------------------------------------------

✅ red_spike ไม่ติด MNF ที่ดิ่งลง (เอา Math.abs ออก)
✅ กรอง sparse node (null > 60%) ออกจาก alert
✅ กรอง offline node (7 วันล่าสุด null) ออกจาก alert
✅ Cron รัน 2 รอบต่อวัน (10:00 + 14:00 BKK) — fallback
✅ Retry logic: login 3 ครั้ง + แต่ละสาขา 3 ครั้ง
✅ yesterday คำนวณด้วย Bangkok timezone
✅ Sync status badge มุมขวาบน (สำเร็จ/ยังไม่ sync/ค้าง X วัน)


----------------------------------------------------------------
9. แนวทางพัฒนาต่อ (ยังไม่ได้ทำ)
----------------------------------------------------------------

ระยะสั้น:
    - แก้ consecutive reset: ให้ "tolerate" gap 1-2 วันแทนที่จะ reset ทันที
      เช่น ถ้า yellow 2 วัน → green 1 วัน → yellow 2 วัน → ให้นับต่อเนื่อง

ระยะกลาง:
    - เพิ่ม secondary baseline (เปรียบกับค่าเฉลี่ยช่วงเดือนเดียวกันปีก่อน)
      เพื่อแก้ปัญหา boiling frog
    - threshold แบบ absolute: ถ้า MNF เกิน X m³/hr เลย ก็ alert ด้วย
      ไม่ใช่แค่ relative diff%

ระยะยาว:
    - Line notify เมื่อมี node แดงใหม่
    - ปุ่ม "รับทราบ" + หมายเหตุ สร้าง audit trail
    - คำนวณ estimated water loss จาก spike
