# อธิบาย EMA Algorithm สำหรับ MNF Monitor
## (Exponential Moving Average สำหรับติดตาม Minimum Night Flow)

---

## 1. EMA คืออะไร และทำไมต้องใช้?

**MNF (Minimum Night Flow)** คือค่าการไหลของน้ำช่วงกลางดึก (02:00–04:00 น.)
ซึ่งเป็นช่วงที่คนใช้น้ำน้อยที่สุด ถ้า MNF สูงผิดปกติ = น้ำรั่วในระบบ

ปัญหาของการดูค่า MNF แบบดิบๆ คือ **มันขึ้นลงตามฤดูกาล ตามวันหยุด ตามแรงดัน**
เปรียบเทียบกับวันก่อนตรงๆ ไม่ได้ เพราะ "ปกติ" ของแต่ละ Node ไม่เท่ากัน

**EMA แก้ปัญหานี้** โดยสร้าง "ค่าเฉลี่ยที่ปรับตัวตามเวลา"
แทนที่จะเปรียบกับค่าเมื่อวาน ให้เปรียบกับ "ค่าเฉลี่ยของ Node นี้ในช่วงที่ผ่านมา"

---

## 2. ที่มาของตัวเลข 0.1333 และ 0.8667

ระบบนี้ใช้ **EMA Period = 14 วัน**

สูตรคำนวณ Multiplier (ตัวคูณ):

```
Multiplier (α) = 2 / (Period + 1)
              = 2 / (14 + 1)
              = 2 / 15
              = 0.13333...
```

จาก α ตรงนี้ได้ตัวเลข 2 ตัว:

```
α       = 0.1333  →  "น้ำหนักของวันนี้"
(1 - α) = 0.8667  →  "น้ำหนักของ EMA เมื่อวาน"
```

**ความหมาย:**
- วันนี้มีน้ำหนักแค่ **13.33%**
- EMA เมื่อวาน (ที่สะสมมา 14 วัน) มีน้ำหนัก **86.67%**

ผลคือ EMA **"เชื่อง"** — ไม่กระโดดตามค่าวันเดียว
ต้องมีค่าสูงต่อเนื่องหลายวันถึงจะดึง EMA ขึ้นได้

### ทำไมต้องเลือก Period = 14?

| Period | α (น้ำหนักวันนี้) | ลักษณะ |
|--------|-----------------|--------|
| 7 วัน  | 0.25 (25%) | ไว — ตอบสนองเร็ว แต่ noise มาก |
| **14 วัน** | **0.1333 (13%)** | **สมดุล — เลือกใช้** |
| 30 วัน | 0.065 (6.5%) | เชื่อง — ตอบสนองช้าเกินไป |

---

## 3. สูตร EMA เต็มๆ

```
EMA วันนี้ = MNF วันนี้ × α + EMA เมื่อวาน × (1 − α)

           = MNF วันนี้ × 0.1333 + EMA เมื่อวาน × 0.8667
```

### ตัวอย่างคำนวณจริง (Node MM-01, Branch 29, วันที่ 22 พ.ค. 2569)

```
ข้อมูลจริงจาก DB:
  MNF วันที่ 22 พ.ค.   = 149.99 m³/hr
  EMA วันที่ 21 พ.ค.   = 154.75 m³/hr  (ค่า EMA ที่สะสมมา)

คำนวณ:
  EMA วันที่ 22 พ.ค. = 149.99 × 0.1333  +  154.75 × 0.8667
                     = 19.99             +  134.12
                     = 154.11 m³/hr  ✓  (ตรงกับ DB)
```

---

## 4. สูตร diff% (ตัววัดความผิดปกติ)

```
diff% = (MNF วันนี้ − EMA เมื่อวาน) / EMA เมื่อวาน × 100
```

**สำคัญ:** เทียบกับ **EMA เมื่อวาน** ไม่ใช่ EMA วันนี้
เพราะ EMA วันนี้ถูกดึงด้วย MNF วันนี้แล้ว การเทียบกับ EMA เมื่อวานจึงสะท้อน "ความเบี่ยงเบน" ได้แม่นกว่า

### ตัวอย่าง (22 พ.ค. 2569):

```
diff% = (149.99 − 154.75) / 154.75 × 100
      = -4.76 / 154.75 × 100
      = -3.07%  ✓  (ตรงกับ DB)

ความหมาย: MNF วันนี้ต่ำกว่า baseline 3.07% → ปกติมาก
```

---

## 5. ระดับ Alert

| สถานะ | เงื่อนไข | ความหมาย |
|-------|---------|---------|
| 🟢 green | diff% < 50% | ปกติ |
| 🟡 yellow | diff% ≥ 50% แต่ < 3 วันติดกัน | เริ่มสูง เฝ้าระวัง |
| 🔴 red_accumulated | diff% ≥ 50% ติดกัน ≥ 3 วัน | รั่วเรื้อรัง |
| 🔴 red_spike | diff% ≥ 200% (วันเดียว) | ท่อแตก เร่งด่วน |

### ทำไม Threshold ถึง 50%?

เพราะข้อมูลจริง (ต.ค. 68 – พ.ค. 69) พบว่า
Node ที่ปกติที่สุดก็ยังขึ้นลงวันต่อวัน **±20–30%** จาก EMA
ถ้าตั้ง threshold ที่ 10% จะ alert ทุกวัน ทุก Node → ไม่มีประโยชน์

## 8. Node Classification — ทำไมบาง Node ถูกข้าม

ก่อน compute EMA ระบบจะตรวจ Node ก่อน 3 ประเภท:

### tiny
```
เงื่อนไข: median ของ MNF < 1.0 m³/hr
เหตุผล:   Node เล็กเกินไป แค่คนเปิดน้ำ 1 บ้านก็ทำ diff% พุ่งได้
การจัดการ: ข้าม ไม่ alert
```

### bimodal
```
เงื่อนไข: Q1 < median × 10%
เหตุผล:   Node มีช่วงไหลปกติ สลับกับช่วงที่ไหลเกือบ 0 (วาล์วปิด/logger ดับ)
           EMA จะถูกดึงลงต่ำในช่วงที่ 0 → alert ผิดพลาดตอนกลับมาปกติ
การจัดการ: ข้าม ไม่ alert
```

### normal
```
เงื่อนไข: ไม่ใช่ tiny และไม่ใช่ bimodal
การจัดการ: คำนวณ EMA และ alert ตามปกติ
```

## 9. Warm-up Period 60 วัน คืออะไร?

เมื่อเริ่ม compute EMA ครั้งแรก ระบบจะดึงข้อมูลย้อนหลัง 60 วัน
**ก่อน** ช่วงที่ต้องการ เพื่อ "อุ่น" EMA ก่อน

**ทำไม?** ถ้า initialize EMA ด้วยวันแรกเพียงวันเดียว
EMA จะใช้เวลาหลายสัปดาห์กว่าจะ "settle" เป็นค่าที่แม่น
การ warm-up 60 วันช่วยให้ EMA เริ่มต้น stable ทันที

```
timeline:
[--- warm-up 60 วัน (คำนวณ EMA แต่ไม่บันทึก) ---][--- ช่วงที่ต้องการ (บันทึก) ---]
      ต.ค. 68                                          ธ.ค. 68 → พ.ค. 69
```

## 10. Cron อัตโนมัติ

```
ทุกวัน 08:00 น. (Bangkok) = 01:00 UTC
  ↓
ดึง MNF เมื่อวาน จาก DMAMA API → upsert mnf_daily
  ↓
recompute EMA ทั้งเดือนปัจจุบัน → upsert mnf_ema_daily
  ↓
mnf_ema_latest view อัปเดตอัตโนมัติ
  ↓
หน้า MNF Monitor แสดงข้อมูลใหม่
```

**หมายเหตุ:** Cron recompute ทั้งเดือน (ไม่ใช่แค่เมื่อวาน)
เพราะต้องการ sequence ต่อเนื่องเพื่อนับ consecutive_count ให้ถูกต้อง
